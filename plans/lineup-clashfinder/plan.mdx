# Lineup Clashfinder

A single-page React app that turns `Lineup.json` (408 performances, 15 stages, 3 days) into a festival timetable you can plan against. The grid plots **time against stage**, one table per day, and can **flip between Horizontal and Vertical** orientation. You pick the sets you want, the app **flags clashes** between selected sets without preventing them, you can **annotate** each set and the plan as a whole, and the entire view is **encoded into the URL** so a link restores it exactly — no backend, no account.

The app is branded **Tomorrowland 2026 W1** and is **mobile-first as well as desktop** (see the responsive decision). It builds directly on the Vite + React + TypeScript app already scaffolded in this repo (strict TS, ESLint, Prettier) and adds one runtime dependency — `lz-string` — for compact URL encoding; everything else is app code.

</RichText>

<Callout id="data-facts" tone="info">

**Facts that shape the build, derived from the file:** 408 performances · 15 stages · 3 days (`FRIDAY`/`SATURDAY`/`SUNDAY`, 17–19 Jul 2026) · times are all `+02:00` · **every record has a valid `startTime`/`endTime`** (no missing-time / NaN path to handle) · 26 sets run past midnight (only the `endTime` crosses to the next calendar day) · up to 3 artists per set · 7 `"More to be announced"` placeholders that carry real times and render like any other set. The `day`/`date` field — not the clock date — is the authoritative day bucket, so an overnight Friday set stays on the Friday table.

</Callout>

<RichText id="scope">

## Objective & scope

**Done means:** loading the app shows Saturday's grid (or whatever the URL encodes); I can switch days, flip orientation, click sets to build _My Lineup_, see overlaps among selected sets outlined, add per-set and plan-level notes, and copy a link that reopens the identical state in a fresh browser with no network.

**In scope:** the timetable grid (both orientations), day tabs, selection + focus toggle, clash detection, notes, and URL-hash state with a Copy-link affordance.

**Out of scope:** any server, persistence beyond the URL (no DB/login), editing the lineup data, search/filter by artist (a fast follow), and mobile-specific layouts beyond responsive grid behavior.

</RichText>

<RichText id="decisions-heading">

## Key decisions

These are the bets that are expensive to change once links are shared, so they are settled here.

</RichText>

<Callout id="decision-url" tone="decision">

**State lives in `location.hash`, compressed with `lz-string`.** A single state object is JSON-stringified and run through `compressToEncodedURIComponent`, then stored after `#`. The hash is chosen over the query string because the fragment is **never sent to a server** (true serverless sharing, nothing in access logs) and survives static hosting. Selections are stored as raw performance `id`s (stable if the lineup file is regenerated) rather than array indices; `lz-string` collapses the repetitive 10-digit ids efficiently. A `v` field versions the schema so old links can be migrated.

</Callout>

<Callout id="decision-time" tone="decision">

**The `date`/`day` field is authoritative for the day-table; positions anchor to it.** A set dated `2026-07-18` belongs to Saturday's table even though its `endTime` rolls into the 19th — bucket by the field, not the clock date (verified: `startTime`'s date always equals the `date` field; only the 26 overnight `endTime`s differ). Within a day, a set's minute offset is its wall-clock `HH:MM` plus `1440 ×` the calendar-day gap between that timestamp and the set's own `date`, so an overnight end lands at e.g. `1470` (24:30) and extends the same day's axis. Reading the string directly (never `new Date()` localization) keeps the layout identical in every viewer's timezone. Seconds are dropped and intervals are half-open `[start, end)`: the data ends sets at `:01`, so without truncation a set ending `14:00:01` would falsely overlap the next set starting `14:00:00` — at minute granularity those 348 boundary touches collapse to **3** genuine same-stage overlaps.

</Callout>

<Callout id="decision-conflict" tone="decision">

**A clash is a time-overlap between two _selected_ sets in the same day, on any stage.** _My Lineup_ spans all three days, but clash detection runs **per day bucket** — sets on different days can't clash, and a Friday overnight set offset past 24:00 must not be compared against a Saturday set at the same wall clock. Within a day: sort selected by start, then for each set scan only the later sets that begin before it ends (an early `break` makes this linear for real lineups, not a full pairwise scan). Overlaps are surfaced — outline on the blocks, the active day's count in the toolbar, a banner in the detail popover — but never block selection, since keeping conflicting picks is the point.

</Callout>

<Callout id="decision-mobile" tone="decision">

**Works on a phone, not just desktop.** The 15-stage grid can't fit a phone width, so on small screens the app defaults to **Vertical** orientation: the time axis pins to the left, stages scroll horizontally, and the toolbar collapses into a **sticky bottom bar** (Flip / Focus / Share). The note and share surfaces become full-width sheets instead of popovers. The Horizontal/Vertical toggle still works on mobile. This is CSS only (container/media queries) over the same `(offsetMinutes, stageIndex)` geometry — there is no separate mobile rendering path.

</Callout>

<Callout id="decision-flip" tone="decision">

**One layout model, two orientations.** Position is always computed as `(offsetMinutes, stageIndex)`. Horizontal maps minutes→X and stage→row; Vertical maps minutes→Y and stage→column. The flip is a CSS-axis swap over the same computed geometry, not a second rendering path — so selection, notes, and clashes behave identically in both. A stage's track is normally a single lane, but the data has **3 genuine same-stage overlaps** (all Friday), so the track packs overlapping sets into sub-lanes (a thin second row/column) rather than letting blocks collide.

</Callout>

<Callout id="risk-urllen" tone="risk">

**URL length is the main risk.** Many selected sets plus long notes inflate the hash. `lz-string` keeps a typical lineup (a few dozen picks, short notes) well under common limits, but free-text notes are unbounded. Per the resolved decision, everything stays in the hash and links are allowed to grow; Copy-link **warns** when the URL passes ~8k chars rather than truncating or moving notes off the link.

</Callout>

## Verification

</RichText>

<RichText id="verify-body">

End-to-end, in the browser (the workflow, not just types):

1. `npm run dev` → Saturday grid renders 15 stages with real act names; `npm run build` and `npm run lint` stay clean.
2. Click several sets including two that overlap (e.g. Space 92 16:30–18:00 and Dino Lenny 16:30–18:00) → both show the clash outline and the toolbar shows "2 clashes", but both stay selected.
3. Flip Horizontal ↔ Vertical → the same picks, notes, and clashes appear with axes swapped.
4. Add a per-set note and a plan note; switch day to Friday and confirm an overnight set (e.g. Franky Rizardo 23:00→00:30) sits at the late end of Friday, not on Saturday.
5. Copy the link, open it in a fresh private window with **no network** → the exact day, orientation, selection, clashes, and both notes restore.
6. Shrink to a phone viewport (~390px) → the grid switches to the vertical, horizontally scrollable layout with the time axis pinned and a sticky bottom bar (Flip / Focus / Share); selection and the clash outline still work by tap.

</RichText>

## Resolved decisions

</RichText>

<Callout id="resolved" tone="decision">

The previously open choices are settled — there are no remaining open questions:

- **Long shared links:** keep all state in the URL hash and allow links to grow; Copy-link **warns past ~8k chars** rather than truncating or moving notes off the link.
- **Unselected sets:** the **full grid shows by default (Focus off)**; Focus hides unselected sets on demand.
- **Branding / title:** the app is **Tomorrowland 2026 W1**.

</Callout>
